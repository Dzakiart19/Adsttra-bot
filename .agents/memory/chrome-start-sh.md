## Rule
`start.sh` MUST be fully dynamic — never hardcode `/nix/store/HASH-...` paths.
Hashes change with every Nix package update; a hardcoded hash from one environment breaks all other environments.

## Critical: mesa-libgbm is now merged into mesa (Nix stable-25_05)
As of 2026-07-23 (Nix channel `stable-25_05`), there is NO separate `mesa-libgbm` package.
GBM (`libgbm.so.1`) is shipped inside `mesa-25.0.7/lib`.
`REPLIT_LD_LIBRARY_PATH` already contains `mesa-25.0.7/lib` → GBM is automatically included.

**Old assumption (WRONG now):** mesa-libgbm is a separate package NOT in REPLIT_LD_LIBRARY_PATH.
**Current reality:** mesa-25.0.7/lib contains libgbm.so.1 AND is in REPLIT_LD_LIBRARY_PATH.

The env.json GBM regex `r'/nix/store/[a-z0-9]+-mesa-libgbm[^":/\s]*'` will return EMPTY
on stable-25_05 — this is OK because GBM is already in BASE_LD via REPLIT_LD_LIBRARY_PATH.

## LD_LIBRARY_PATH resolution order (current)
1. **`$REPLIT_LD_LIBRARY_PATH` env var** — set by Nix at dev runtime and deployment BUILD time.
   Contains all needed libs including mesa-25.0.7 (with GBM). This is the primary source.
2. **`.nix_env` fallback** — if REPLIT_LD_LIBRARY_PATH not set at runtime (autoscale containers),
   read from `.nix_env` (format: `LD_PATHS|`, GBM part empty). Generated by build command.
3. **env.json** — `.cache/replit/nix/env.json` for GBM_LIB (may return empty on stable-25_05).
   Only used as supplemental GBM source; does NOT replace step 1/2.

## .nix_env — how it's generated (UPDATED)
Build command in `.replit` now uses:
```bash
printf '%s|' "${REPLIT_LD_LIBRARY_PATH}" > .nix_env
```
- Captures `REPLIT_LD_LIBRARY_PATH` directly during BUILD (available in Replit build containers)
- Format: `LD_PATHS|` (trailing pipe, empty GBM since mesa includes it)
- start.sh reads: `IFS='|' read -r _nix_ld _nix_gbm < .nix_env` → BASE_LD=_nix_ld ✓
- Gitignored; regenerated on every publish/build

**Why old approach (Python + env.json) failed in production:**
Python script read env.json which may not exist in autoscale build containers →
.nix_env saved with empty values → REPLIT_LD_LIBRARY_PATH not set at runtime →
LD_LIBRARY_PATH empty → `libgbm.so.1: cannot open shared object file` → Chrome exit 127.

## Chrome binary resolution order
1. `.puppeteer_cache/` local dir — installed by deployment build command (`PUPPETEER_CACHE_DIR=.puppeteer_cache npx puppeteer browsers install chrome-headless-shell`)
2. `/home/runner/.cache/puppeteer/` — puppeteer default download location
3. Auto-install — if nothing found, run `PUPPETEER_CACHE_DIR=.puppeteer_cache npx puppeteer browsers install chrome-headless-shell`

`.puppeteerrc.cjs` also directs puppeteer's own cache lookup to `.puppeteer_cache/` (project-local).

**Never** `find /nix/store` — causes timeout. Only search small local dirs.

## Why
Nix store hashes change when packages are updated. Hardcoded paths from one session break
any other environment (fresh clone, re-publish, Nix update). The dynamic approach works
because REPLIT_LD_LIBRARY_PATH is always set correctly by Replit's Nix loader for the
current environment — both in dev and in deployment build containers.
